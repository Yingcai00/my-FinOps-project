## 1. 对当前项目的理解（通读结论）

### 1.1 仓库形态

- **前端**：大量静态 HTML 原型（统一侧栏、卡片 + 表格 + 弹窗），覆盖成本总览、成本池、定价、账单（多 Tab）、分账、B 类资产、用量确认（PC/移动/企微推送）、**用量管理**、系统管理等。
- **规格**：已存在高粒度的**账单定稿**（账期 + 版本、用量确认闸、发送后只读、导出一律 PDF）与 **RBAC/多租户** 产品规格，质量高于普通原型说明。
- **后端**：`FINOPS_BACKEND_ARCHITECTURE.md` 描述 Spring Boot 单体、成本 / 产品 / 计费 / 账单 / 配置等域；与前端原型尚未一一绑定接口。
- **张力点**：`FinOps产品方案文档_v1.0.md` 中 P2 仍写「账单多格式导出」，与已定稿的 `bill-management` 规格（**账单类导出仅 PDF**）不一致。**公司内部实施应以已定稿规格为准**，并回写产品方案索引，避免实施团队分叉。

### 1.2 业务链路（已在文档与页面中体现）

资产与成本主数据 → 成本池与定价规则 → 用量 / 计量 → 分摊与分账标签 → 部门 / 服务 / 资源账单 → 用量确认与关账 → 定稿版本与对外发送 → 分析与管理端重跑。

---

## 2. 问题陈述（公司内部视角）

**要解决的问题**：在单一组织（或「一租户多 BU」）内，把云与私有资源的真实成本、内部结算价、用量与账单串成**可信、可审计、可关账**的一条链；减少 Excel 影子流程；让财务与业务对「同一套数字」达成一致。

**非目标（首版建议明确写进立项）**：多云公有云账单全自动拉取与复杂价差引擎；以 Excel 作为账单定稿载体；发送后界面内改账。

## 4. 需求证据与现状差距（内部诊断）

| 能力 | 规格 / 原型现状 | 对内落地含义 |
|------|-----------------|---------------|
| 关账与定稿 | 规格完整 | 后端必须实现 `BillPeriod` / `BillVersion`、确认状态机、发送事件与只读闸 |
| 部门 → 服务 → 资源下钻 | 规格 + `bill-management.html` 方向 | 前后端契约需固定面包屑与筛选传递 |
| 分账标签 | `allocation-management.html` + 规格 | 发送前可写、发送后冻结；不得静默触发全量重算 |
| 用量管理 | `usage-management.html` | 与计量源（玄武等）对齐：每日上报、补录、删改申诉策略需后端与消息/工单 |
| 用量确认 | `usage-confirm-*.html` + 企微设置 | 与关账闸联动；权限见 RBAC 规格 |
| RBAC | `2026-05-08-finops-rbac-*` | 服务端强制数据范围；侧栏裁剪仅为 UX |

## 5. 方案比选（至少两种）

### 方案 A：规格优先的「关账账单内核」先行（推荐）

- **摘要**：先实现账单域 + 用量确认 + 版本化 PDF + 权限模型；成本 / 定价以「能算出账」的最小闭环接入。
- **工作量**：中（团队按模块并行时约 8～12 周量级，视人力与存量系统接口而定）。
- **风险**：中。外部计量与组织主数据接口延迟会挤压联调。
- **复用**：直接对齐已有 `bill-management`、`resource-bill-detail`、`usage-management`、用量确认原型与两份 2026-05-08 规格。

### 方案 B：「成本与定价运营平台」先行

- **摘要**：优先做成本池、定价、资产台账与报表；账单以「预览稿 / 非定稿」形式存在，关账闸后补。
- **工作量**：中高（易陷入长期「可对账但未定稿」状态）。
- **风险**：高。财务与部门对「数字何时算数」易产生分歧，后期补关闸改造成本大。
- **适用**：仅当公司当前最大痛点是成本主数据与定价，且短期无对内出账压力。

### 方案 C：双轨并行（不推荐）

- **摘要**：A 与 B 两条线同时全量铺开。
- **风险**：高。集成点与责任边界模糊，易重复建模。

**推荐**：**方案 A**。理由：贵司已有高质量账单定稿与 RBAC 规格，且原型已覆盖关键界面；对内 FinOps 的「可信」来自关账与版本，而不是更多报表类型。

## 6. 推荐目标架构（公司内部）

### 6.1 应用分层

1. **门户与导航**：按角色裁剪菜单（与 RBAC 规格一致）；统一账期 / 账单版本上下文组件。
2. **运营配置域**：组织、用户角色、成本池、定价规则、分账标签字典、系统参数（出账日、确认截止日）。
3. **计量与用量域**：源系统同步任务、用量台账、`usage-management` 对应的补录与申诉策略、对账任务。
4. **计费与账单域**：计费运行、分摊、账单聚合、版本、PDF 渲染、发送记录、管理端重跑。
5. **确认与协作域**：用量确认流程、企微（或现有 IM）推送、异议与驳回闭环。
6. **分析与优化域**：`bill-overview` / `cost-analysis` 类只读分析；默认不改变定稿数字，可做「相对最新定稿版本的模拟」若未来需要（首版可不建）。

### 6.2 与现有后端文档的映射

在 `FINOPS_BACKEND_ARCHITECTURE.md` 的模块上**显式增加**：

- `BillVersionService`（生成、重跑、只读校验）
- `UsageConfirmationService`（关账闸查询）
- `PdfExportService`（模板 + 版本水印）
- `AllocationTagWriteGate`（发送前可写 / 发送后拒绝）

数据表在现有 `usage_records`、账单相关表基础上扩展 `bill_version`、`usage_confirmation` 等（命名以实现为准，概念与 `bill-management` 规格 §5 对齐）。

### 6.3 集成边界（内部常见系统）

| 系统 | 方向 | 要点 |
|------|------|------|
| 资产 / CMDB / 玄武类 | 入 | 实例 ID、规格、归属、生命周期；需幂等与延迟补偿 |
| 监控 / 计量源 | 入 | 与用量管理「每日上报」一致；缺口走补录 |
| 组织架构（HR / AD） | 入 | 部门路径与负责人；驱动数据权限 |
| ERP / 总账 | 出 | 建议仅传递「已定稿版本」汇总凭证维度字段；明细在 FinOps 内 PDF 留存 |
| 企微 / OA | 出 / 回调 | 确认与审批链接；回写确认状态 |

## 7. 信息架构（菜单与里程碑）

在现有原型菜单基础上，公司内部首版建议**固定**以下「可演示闭环」顺序：

1. 系统管理 + RBAC 配置（含租户内功能项收紧）
2. 组织与资源主数据（B 类资产或等价）
3. 成本池与定价（能锁定单价来源）
4. **用量管理** + 计量同步任务监控
5. 分账标签维护（在用量确认关账前完成策略宣导）
6. 账单详情（账期 + 版本 + 下钻 + PDF）
7. 用量确认（含截止提示）→ 发送 → 只读
8. 账单分析（只读）

---

## 8. 成功标准（验收口径摘要）

1. 任意用户在权限外组织下**接口与导出均为空**（不仅前端隐藏）。
2. 未满足用量确认闸时：**不能**将账单标记为已发送，**不能**生成对外 PDF 正式版（或生成明确标注「预览」的水印样式，二选一写死）。
3. 已发送账单：**任意字段**不可 API 修改；修正仅通过新版本重跑；旧版本可只读查询与同版本 PDF。
4. 账单域导出符合 `2026-05-08-bill-management-drilldown-design.md` 的 PDF 约束。
5. 用量管理行为符合 `usage-management.html` 所表达的业务规则（补录无申诉、删改触发提醒与批量能力由后端执行）。

## 9. 开放问题（需内部工作坊拍板）

1. **多 BU 是否共租户**：部门路径是否需强制二级部门 + BU 节点（规格已留扩展口）。
2. **重跑后是否必须重新走用量确认**：规格建议「是」；若否需书面承担对账风险。
3. **分析类导出**：是否在 RBAC 中单列为「受控 CSV」（与「账单 PDF」区分）；若允许，模板与水印策略是什么。
4. **ERP 凭证粒度**：到部门、到成本中心、还是到项目。
