# 云管系统 · 服务 / 规格簇 / 规格 排序功能需求文档

> 版本：v1.0 | 日期：2026-05-07 | 状态：评审中

---

## 1. 文档概述

### 1.1 背景

云管系统的服务目录由**三层结构**组成：

```
服务（Service）
  └── 规格簇（SpecCluster）
        └── 规格（Spec）
```

服务、规格簇、规格目前均支持新增与编辑。服务类型编辑弹窗已存在"排序"字段（默认值 100），规格簇与规格尚无排序字段。

用户在查看规格列表时，需要**可控的展示顺序**，以便将常用或重要规格优先展示给申请者。

### 1.2 目标

1. 为服务、规格簇、规格三层均提供**手动排序字段**，支持在编辑时调整展示顺序。
2. 规格列表的全局排序规则：**服务排序 → 规格簇排序 → 规格排序**（升序）。
3. 提供友好的排序填写辅助：**默认值自动填入当前维度最大值+10**、**重复值实时提醒**。
4. 明确排序断档（数字不连续）的处理策略，不影响业务使用。

### 1.3 术语

| 术语 | 说明 |
|------|------|
| 排序值 | 正整数，同维度下升序排列，数值越小展示越靠前 |
| 同维度 | 指同一层级且同属父节点的兄弟项（如同一服务下的所有规格簇） |
| 断档 | 同维度内排序值不连续，如 10、20、50（中间跳过 30、40） |
| 重复值 | 同维度内已存在与用户输入相同的排序值 |

## 2. 排序规则设计

### 2.1 全局展示排序规则

规格列表（含定价、计量等下游页面）统一按以下三级键排序（均升序）：

```
ORDER BY service.sort_order ASC,
         spec_cluster.sort_order ASC,
         spec.sort_order ASC
```

> **注意**：当排序值相同时，以**创建时间**作为次级排序，保证列表稳定。

### 2.2 同维度排序范围定义

| 层级 | 同维度定义 |
|------|----------|
| 服务 | 系统全局所有服务 |
| 规格簇 | 同一服务下的所有规格簇 |
| 规格 | 同一规格簇下的所有规格 |

## 4. 排序断档问题分析

### 4.1 断档定义

排序断档指同维度下的排序值不连续，例如：

```
规格簇 A：sort_order = 10
规格簇 B：sort_order = 20
规格簇 C：sort_order = 50   ← 与B之间存在断档（跳过30、40）
```

### 4.2 断档产生的原因

| 场景 | 说明 |
|------|------|
| 增量追加 | 使用"最大值+10"默认策略后，删除中间项不回收编号 |
| 人工跳跃设置 | 用户主动设置非连续值（如刻意留空间供后续插入） |
| 批量导入/迁移 | 历史数据排序值不连续 |
| 删除后不重排 | 删除排序值 30 的项后，10/20/40/50 之间出现断档 |

### 4.3 断档是否有影响

**结论：断档不影响业务功能，展示顺序仅取决于排序值的相对大小，不要求连续。**

| 维度 | 影响说明 |
|------|----------|
| 展示顺序 | ❌ 无影响。排序依赖大小比较，10、20、50 的相对顺序与 1、2、3 等价 |
| 用户体验 | ⚠️ 轻微影响。用户看到排序值"跳跃"时可能困惑，但不影响使用 |
| 插入灵活性 | ✅ 有利。有断档意味着中间仍有可插入空间，反而是**刻意留断档的好处** |
| 数据存储 | ❌ 无影响 |
| 系统性能 | ❌ 无影响 |

### 4.4 推荐：使用"步长10"策略主动留断档

为了降低断档带来的用户困惑，同时保留插入灵活性，推荐**默认步长为10**：

```
初始规格簇：10、20、30
插入新规格簇到第2位：设置排序值 15 → 顺序变为 10、15、20、30
无需批量更新其他项
```

此策略的好处：
- 插入新项只需设置中间值，无需重排其他项
- 步长内有充足空间供插入（0~9之间可插入9个值）
- 即使出现断档，顺序语义依然清晰

### 4.5 避免断档的措施（可选，不强制）

如果产品希望排序值尽量整洁，可在以下时机提供**可选的"整理排序"功能**（非本期必选）：

| 措施 | 实现方式 | 触发时机 |
|------|----------|----------|
| 删除后自动重排 | 删除某项后，将同维度其余项按当前顺序重排为 10/20/30... | 删除操作后台执行 |
| 手动整理排序 | 提供"整理排序"按钮，将当前维度项按现有顺序重置为 10/20/30... | 用户主动触发 |
| 拖拽排序 | 支持列表内拖拽调整顺序，后台自动按位置重新计算排序值 | 拖拽操作后自动更新 |

> **本期建议**：先实现填写辅助（默认值+重复提醒），不强制整理，由用户按需整理。

## 5. 交互细节说明

### 5.1 默认值逻辑

```
新建时：default_sort = max(同维度所有已有排序值) + 10
       若无已有项则：default_sort = 10
编辑时：default_sort = 当前项的排序值（回显）
```

### 5.2 最大值提示展示位置

```
┌────────────────────────────────────────────┐
│  排序  [ 30         ↑↓ ]   当前最大排序值：20  │
│         ⚠ 当前已存在排序值 20，保存后将与       │
│           「标准通用服务器」顺序相邻，建议调整   │
└────────────────────────────────────────────┘
```

- 最大值提示：灰色辅助文案，始终展示
- 重复警告：橙色警告文案，仅在检测到重复时展示
- 警告不阻止保存（提示性，非拦截性），用户可以知情后确认保存

### 5.3 重复值的业务语义

重复排序值在展示时，会与另一个同排序值的项**顺序不确定**（由创建时间决定），因此警告文案中明确提示用户后果，便于用户决策是否调整。

### 5.4 后端存储

- 三张表均增加 `sort_order` 整型字段（DEFAULT 值建议 10 或 0）
- 查询接口返回时按 `sort_order ASC, created_at ASC` 排序
- 前端调用保存接口时传入 `sort_order` 字段值

## 6. 非功能需求

| 编号 | 类别 | 描述 |
|------|------|------|
| NFR-01 | 数据一致性 | 重复排序值不做拦截，允许存在，后端不设唯一索引 |
| NFR-02 | 向后兼容 | 存量数据中若 sort_order 为 NULL，展示时排在最后（ORDER BY sort_order IS NULL DESC, sort_order ASC） |
| NFR-03 | 性能 | 最大值查询应在弹窗打开时执行，不影响列表页加载 |

## 7. 验收标准

| 编号 | 验收点 |
|------|--------|
| AC-01 | 新建服务时，排序默认值 = 现有最大值 + 10 |
| AC-02 | 新建规格簇时，排序默认值 = 同服务下最大值 + 10 |
| AC-03 | 新建规格时，排序默认值 = 同规格簇下最大值 + 10 |
| AC-04 | 输入框旁展示"当前最大排序值：XX"，数字准确 |
| AC-05 | 输入与已有项重复时，弹出橙色重复提示（含被重复项名称） |
| AC-06 | 规格列表按服务排序→规格簇排序→规格排序三级升序展示 |
| AC-07 | 排序断档时（如10、20、50），列表顺序仍正确，不出现乱序 |
| AC-08 | 存量 sort_order 为空的数据排在最后 |
| AC-09 | 排序值不允许为空、小数、负数、非数字 |

## 8. 迭代建议（本期不做）

- **拖拽排序**：在服务管理列表、规格簇列表、规格列表支持行拖拽快速调序
- **批量整理排序**：提供一键将同维度排序值重置为 10/20/30... 的操作
- **排序值可视化**：在列表页展示排序值列，方便用户无需进入编辑即可查看当前顺序
